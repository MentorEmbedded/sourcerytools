<P><FONT SIZE=2>I hate using these enums because different compilers seem to</FONT>
<BR><FONT SIZE=2>treat them differently.  An alternative would be to define a</FONT>
<BR><FONT SIZE=2>static bool "wildcard" in these classes, set it to "true" or "false",</FONT>
<BR><FONT SIZE=2>and then pass that as the template argument.  But as I recall,</FONT>
<BR><FONT SIZE=2>there was some problem with the Metrowerks compiler that</FONT>
<BR><FONT SIZE=2>made this not a viable option.  </FONT>
</P>

<P><FONT SIZE=2>I guess I'd be surprised if Metrowerks had a problem with this - we use exactly this sort of thing in ElementProperties. Moreover...</FONT></P>

<P><FONT SIZE=2>I guess I am wondering if that</FONT>
<BR><FONT SIZE=2>problem still exists or whether it would be OK to overhaul</FONT>
<BR><FONT SIZE=2>the Domain classes to use static data types instead of enums.</FONT>
<BR><FONT SIZE=2>The one drawback to this approach that I see is you have to</FONT>
<BR><FONT SIZE=2>add some compiled source file that initializes these static</FONT>
<BR><FONT SIZE=2>variables.  You can't just stick this in the class definition.</FONT>
</P>

<P><FONT SIZE=2>Yes you can. The compilers we're using have supported this for quite some time. You just say:</FONT>
</P>

<P><FONT SIZE=2>  template <class T></FONT>
<BR><FONT SIZE=2>  struct Foo</FONT>
<BR><FONT SIZE=2>  {</FONT>
<BR><FONT SIZE=2>    static const bool wildcard = true;</FONT>
<BR><FONT SIZE=2>  };</FONT>

<P><FONT SIZE=2>On the other hand, if you think the aCC compiler is full of</FONT>
